**category**   : behavior_changes  
**area**       : tcp_proxy  
**summary**    : tcp_proxy: fix to make tcp_proxy ON_DOWNSTREAM_TLS_HANDSHAKE mode work without max_early_data_bytes (#42505)  
**summary**    : changelog: fix typos in the current changelogs and clean them up (#43161)  
**commit**     : [f1dcefb56bb5921e7f94a6f81522f34ed451e79a](https://github.com/envoyproxy/envoy/commit/f1dcefb56bb5921e7f94a6f81522f34ed451e79a) [f11d3ef8d5573b4685ee3385a23d14f07f5a8405](https://github.com/envoyproxy/envoy/commit/f11d3ef8d5573b4685ee3385a23d14f07f5a8405)  
**pr**         : [42505](https://github.com/envoyproxy/envoy/pull/42505) [43161](https://github.com/envoyproxy/envoy/pull/43161)  
**description**:  
The TCP proxy filter now requires [max_early_data_bytes]() to be explicitly set when using [upstream_connect_mode]() modes other than ``IMMEDIATE`` (i.e., ``ON_DOWNSTREAM_DATA`` or ``ON_DOWNSTREAM_TLS_HANDSHAKE``). The field can be set to ``0`` to disable early data buffering while still using delayed connection modes. Configurations using these modes without ``max_early_data_bytes`` will now fail validation at startup.

**area**       : conn_pool  
**summary**    : conn_pool: add upstream_rq_active_overflow counter (#44164)  
**commit**     : [c2117809b2a44ad619896cb784aa0f53c6b7b493](https://github.com/envoyproxy/envoy/commit/c2117809b2a44ad619896cb784aa0f53c6b7b493)  
**pr**         : [44164](https://github.com/envoyproxy/envoy/pull/44164)  
**description**:  
Added new ``upstream_rq_active_overflow`` counter incremented when a request is rejected due to the ``max_requests`` circuit breaker being exhausted in ``attachStreamToClient()``. Previously this condition incorrectly incremented ``upstream_rq_pending_overflow``, making it impossible to distinguish pending queue saturation from active request saturation via metrics alone. The new counter is now the authoritative signal for this path by default; set runtime flag ``envoy.reloadable_features.skip_pending_overflow_count_on_active_rq`` to ``false`` to also increment ``upstream_rq_pending_overflow`` on this path, preserving backwards compatibility with existing dashboards and alerts.

**category**   : minor_behavior_changes

**summary**    : change log: add change log for previous refactoring (#44142)  
**commit**     : [fc4dc91f4ca9619c99e9fac1252786ad7f773e53](https://github.com/envoyproxy/envoy/commit/fc4dc91f4ca9619c99e9fac1252786ad7f773e53)  
**pr**         : [44142](https://github.com/envoyproxy/envoy/pull/44142)  
**description**:  
The ``route()``, ``clusterInfo()`` and ``virtualHost()`` methods on the HTTP filter callbacks and stream info interfaces now are refactored to return ``OptRef<const T>`` instead of shared pointers. And additional methods ``routeSharedPtr()``, ``clusterInfoSharedPtr()`` and ``virtualHostSharedPtr()`` are added to return shared pointers. The shorter ``route()``, ``clusterInfo()`` and ``virtualHost()`` methods are preferred for most use cases where the returned reference is not used beyond the scope of the current function and the caller does not want to manage shared pointer ownership. The new shared pointer methods are intended for use cases where the caller needs to keep the ownerships to the route, cluster, or virtual host.

Added support for building Envoy with OpenSSL as an alternative to the default BoringSSL. This is achieved through a BoringSSL compatibility layer (``bssl-compat``) that translates BoringSSL API calls to OpenSSL, allowing Envoy's TLS code to remain unchanged. To build with OpenSSL, use ``--config=openssl``. HTTP/3 (QUIC) is disabled for OpenSSL builds. Note that OpenSSL builds are not currently covered by the `Envoy security policy <https://github.com/envoyproxy/envoy/blob/main/SECURITY.md>`_. See :repo:`bazel/SSL.md <bazel/SSL.md>` for details.

**commit**     : [ac8ea19f77a183dac1ee31dcc4b6f2f345765377](https://github.com/envoyproxy/envoy/commit/ac8ea19f77a183dac1ee31dcc4b6f2f345765377)  
**pr**         : [44610](https://github.com/envoyproxy/envoy/pull/44610)  
**description**:  
Added SCONE (Standardized Communication with Network Elements) support for QUIC. This includes a new [enable_scone]() field in ``QuicProtocolOptions``, and support in Envoy Mobile via a new ``enableScone()`` API in ``EngineBuilder``. SCONE bandwidth and timestamp data are now propagated to the application via new ``scone_max_kbps`` and ``scone_timestamp_ms`` fields in the ``envoy_stream_intel`` struct.
